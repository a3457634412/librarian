## 问题 1: 数据模型散落

**现状**: 一篇文章在系统里流转时，字段散落在 tagged.json、.md 文件、索引文件三个地方。后续模块（keyword.py、indexer.py、searcher.py）各自从 .md 用正则反解析，而不是直接读原始数据。同一个 `title` 被 obsidian_writer 写出去又被 keyword.py 用正则抠回来。

**后果**:
- 改一个字段的格式，下游所有解析都得跟着改
- 无法回答"某篇文章现在什么状态"
- .md 是给人看的格式，不是给程序读的

**方案**: 引入 `ArticleStore`

```python
# models.py
@dataclass
class Article:
    id: str           # "2026-05-30::article::MemOS v2..."
    title: str
    url: str
    source: str
    points: int
    tech_tag: str = ""
    tech_summary: str = ""
    trend_signal: str = ""
    relevance_to_me: str = ""
    signal_level: str = ""  # red / yellow / green
    state: str = "ingested"  # ingested → tagged → indexed → archived

class ArticleStore:
    """所有模块读写的唯一入口。底层先 JSON，以后可换 SQLite"""
    def ingest(self, articles): ...
    def get_by_state(self, state): ...
    def get_by_date(self, date): ...
    def update(self, article_id, fields): ...
    def mark_state(self, article_id, new_state): ...
```

**涉及的改动**:
- 新建 `librarian/models.py`（Article + ArticleStore）
- `fetch_sources.py` → ingest 到 store，不再只写 raw.json
- `tagger.py` → 从 store 读 untagged，打完写回 store
- `obsidian_writer.py` → 变成 store 的**导出视图**，不是数据源
- `indexer.py` / `keyword.py` / `searcher.py` → 从 store 读，不再解析 .md
- `agent.py` → 所有步骤通过 store 交换数据

**不改的**: tagger prompt 格式、索引算法、推送逻辑。本问题只管数据流。

## 问题 2: 抓取失败导致文章永久丢失

**现状**: 三路信源（Simon/GitHub/HN）抓取失败时，`fetch_*()` 捕获异常 → `print("⚠️")` → 返回空列表 → 管线继续。当天没抓到的那批文章永远不会再尝试。

**后果**:
- Simon 今天发了一篇核心相关的，网络抽风 3 秒 → 永久错过
- 管线日志显示"0 篇"，无法区分"真的没发文"vs"抓取失败了"

**方案**: 抓取加 retry + pending 机制

```python
# fetch_sources.py
def fetch_with_retry(source_name, max_attempts=3, interval=30):
    for attempt in range(max_attempts):
        try:
            return fetch_fn()
        except Exception:
            if attempt < max_attempts - 1:
                time.sleep(interval)
    # 全失败 → 写 pending
    pending = load_pending()
    pending.append({"source": source_name, "date": today, "attempts_left": 2})
    save_pending(pending)
    return []

def retry_pending():
    """每次 cron 跑之前先补抓 pending 中还没成功的源"""
    pending = load_pending()
    for item in pending:
        result = fetch_with_retry(item["source"])
        if result:
            pending.remove(item)
    save_pending(pending)
```

**涉及的改动**:
- `fetch_sources.py`: 每个 `fetch_*()` 包一层 retry，失败写 `pending_fetches.json`
- `agent.py`: 步骤 1 之前加 `retry_pending()`
- 新建 `librarian/.pending_fetches.json`

**不改的**: 抓取逻辑本身不换，只加错误处理和重试。

## 问题 3: 索引中途崩溃导致数据永久漏掉

**现状**: `indexer.py` 的增量模式用 mtime 判断文件是否已处理。如果一次索引构建中途崩溃（OOM / HF 镜像超时），mtime 已写入 state，但向量没写入 index。下次增量跑时 mtime 匹配 → 跳过 → 永久漏掉。

**后果**: 每天 25 篇文章，索引挂了就少 25 条向量。连续挂 3 天就是 75 条。用户搜不到，不知道有问题。

**方案**: mtime 和向量写入原子化

```python
# indexer.py — 改为先编码、拿到结果、再更新 state
def build_index(mode="--incremental"):
    for each file:
        if unchanged: continue
        entries = process_file(file)           # 1. 提取文本
        vectors = model.encode(entries)        # 2. 编码 ← 这步可能挂
        save_index(index_file, ids, vectors)   # 3. 写入索引
        state[file_path] = mtime               # 4. 索引写入成功后，才更新 mtime
        save_state(state)                      # 5. 持久化 state
```

现在的顺序是把 state 更新放在前面了——改成**先完成所有 I/O 计算，最后才写 mtime**。挂了的话 mtime 没变，下次重新处理。

同时加 retry：
```python
for attempt in range(3):
    try:
        vectors = model.encode(entries)
        break
    except Exception:
        if attempt == 2:
            mark_pending_index(entries)  # 三次都失败，记下来
        time.sleep(30)
```

**涉及的改动**:
- `indexer.py`: 调整写入顺序 + 编码加 retry
- 新建 `librarian/.pending_index.json`（失败的条目清单）

## 问题 4: 搜索不懂问题类型

**现状**: "agent 记忆怎么做"（探索型）和 "MemOS hybrid retrieval 实现"（定位型）走完全相同的检索流程 → 关键词权重 + 语义权重 → 得分排序。前者应该优先返回综述/wiki，后者应该优先返回具体文章细节。

**后果**:
- 探索型问题：排前面的可能是相关性高但太窄的文章，缺全景
- 定位型问题：综述排在前面，具体文章被挤到后面

**方案**: 搜索前加轻量路由

```python
# search/router.py
def classify_query(query: str) -> str:
    """LLM 一句话分类，成本 ~10 token"""
    prompt = f"""判断以下查询的类型，只输出 "broad" 或 "specific"：
查询: "{query}"
- broad: 想了解全貌、方案对比、该怎么做
- specific: 想找具体实现细节、某篇文章、某个数值"""
    return call_llm(prompt, max_tokens=5).strip()

# searcher.py
def search(query):
    q_type = classify_query(query)
    if q_type == "broad":
        # 优先搜 core_index (wiki/综述)，调高 core_weight 到 2.0
        # 返回结果优先展示综述
    else:
        # 优先搜 recent_index，关键词权重调高到 0.5
```

也可以在检索结果上加：
- broad → 结果前置综述，文章作为补充
- specific → 结果前置匹配度最高的单篇，综述不展示

**涉及的改动**:
- 新建 `librarian/search/router.py`（查询分类）
- `librarian/search/semantic.py`: 接受动态权重参数
- `librarian/search/hybrid.py`: 接受动态权重参数
- `librarian/searcher.py`: 先路由再搜索

## 问题 5: Shell 遗留未清理

**现状**: `agent.py` 已经统一了日常管线，但 `process_new.sh` (416行)、`curator.sh`、`collision_detector.sh`、`search_obsidian.sh`、`search_engine.sh` 仍存在。cron 实际跑的是哪个不确定。

**后果**:
- 两套实现不一定行为一致
- 新功能只加在 Python 端，shell 端变成僵尸代码
- 新人看到两个入口会困惑

**方案**: 清理，不迁移

```bash
# 第一步：确认 cron 跑的是 agent.py
# 第二步：删除不再使用的 shell 脚本
# 第三步：保留有用的 shell 作为降级路径或删除

要删的:
  process_new.sh        → processor.py + curator.py + contradiction.py + archiver.py
  curator.sh            → curator.py
  collision_detector.sh → contradiction.py
  search_engine.sh      → search/keyword.py
  hybrid_merge.py       → search/hybrid.py (独立文件 → 模块)

要保留的（仍有独立价值或作为降级路径）:
  search_obsidian.sh    → 简单搜索的 bash 入口，不需要 Python 环境
  archiver.sh           → 独立归档操作
  preemptive_brief.sh   → 主动简报，Python 对应功能还未完全覆盖
  profile.sh            → 搜索画像，Python 对应功能未覆盖
```

**涉及的改动**: 删除文件，确认 cron 指向 `agent.py`
