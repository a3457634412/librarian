# 优化路线图

## 职责分界（明确后不再变动）

```
沈念 (ShenNian)                    资料管理员 (Librarian)
══════════════════════              ══════════════════════
"你是谁"                           "外面发生了什么"
                                    "项目环境是什么样的"

about-me/                          外部信源 → 筛选 → 萃取
  ├─ 兴趣.md                       精华入库 → 碰撞检测
  ├─ 工作.md                       踩坑管理（分类/合并/归档）
  ├─ 偏好.md                       wiki 辅助（提案、不直接写）
  └─ 其他.md
                                   核心层维护
wiki/（沈念只读不写）                ├─ curated/reviews/
对话存档                            ├─ pitfalls/   ← 沈念不再管，归 librarian
状态时间线                           ├─ curated/insights/
                                   └─ 演化时间线

输出: 越来越懂你                    输出: 喂给沈念（外部信号 + 踩坑上下文）
```

## 资料管理员 — 5 个 Phase

### Phase 1 ── 基础设施重构

> 不依赖沈念，独立完成。改的是 librarian 自身的索引和存储架构。

| # | 任务 | 改动 |
|---|------|------|
| 1.1 | **索引分层** | `index_builder.py` → `index_manager.py`，拆成 core_index / recent_index / archive_index 三个 JSON |
| 1.2 | **增量向量化** | 只编码当天新增 .md，旧条目用 id 跳过 |
| 1.3 | **段落级分块** | wiki/综述/踩坑 按 `##` 拆段编码，不再整篇 500 字 |
| 1.4 | **归档机制** | `archiver.sh`：>7 天文章移入 archive/，从 recent_index 移除 |
| 1.5 | **踩坑迁移** | pitfall 文件从沈念迁出，归入 librarian 的核心层管理（路径：`pitfalls/`） |
| 1.6 | **检索适配三层** | `search_semantic.py` + `hybrid_merge.py` 支持分层权重 |

### Phase 2 ── 入库关口

> 依赖 Phase 1（索引结构就绪）。改的是 `fetch_and_tag.sh` 的 prompt 和 `write_to_obsidian.sh` 的写入逻辑。

| # | 任务 | 改动 |
|---|------|------|
| 2.1 | **质量评分** | prompt 加 `quality: 1-5`，≤2 不存 |
| 2.2 | **去重合并** | 同项目同日多篇 → 合并为一条 |
| 2.3 | **信号/噪声分类** | `signal_type`: new_paradigm / incremental / rehashing |
| 2.4 | **萃取 extracted_insight** | 50-80 字：这条对你意味着什么。替代 summary 做向量化文本 |
| 2.5 | **关系判断** | 新数据入库前判断：迭代(更新) / 优化(更新+标旧) / 选择(建问题域文件夹) / 全量替换(整个文件夹重写) |
| 2.6 | **只写精华** | 只存 quality≥3 的文章，raw 入口处理完就封存 |

### Phase 3 ── 知识演化引擎

> 依赖 Phase 2（有 extracted_insight 和关系标记才能做碰撞）。核心智能层。

| # | 任务 | 改动 |
|---|------|------|
| 3.1 | **追踪线程** | `thread_tracker.py`：同技术连续 ≥3 天 → 建立线程，存 `threads.json` |
| 3.2 | **碰撞检测** | `collision_detector.sh`：新文章 vs 核心库 → 冲突/补充/替代 三类标记 |
| 3.3 | **成熟度迁移** | `maturity_migrator.py`：🌱 → 🌿 → 🌳 自动升级，记录 `timeline.json` |
| 3.4 | **策展触发智能化** | `curator.sh`：从固定阈值改为追踪线程活跃 + 碰撞信号触发 |
| 3.5 | **核心层整理** | 定期扫 vault 结构，判断踩坑能否合并、wiki 是否需要新建子目录、选择关系的技术是否该归类 |

### Phase 4 ── 主动输出

> 依赖 Phase 3（有碰撞和追踪数据才能做预警）。改的是 `process_new.sh` 的推送和日报。

| # | 任务 | 改动 |
|---|------|------|
| 4.1 | **异常预警** | 相关文章数 vs 7 日均值、标签分布突变、追踪线程停滞 |
| 4.2 | **盲区提醒** | 标签下 >30 篇但你的交互为 0 → 提示 |
| 4.3 | **日报改版** | 推送内容变为：今日精华 (3-5 条) + 碰撞提醒 + 追踪更新 + 盲区 (每周) |

### Phase 5 ── 对话增强

> 依赖 Phase 1（三层索引就绪）。改的是 `search_obsidian.sh` + `search_engine.sh`。

| # | 任务 | 改动 |
|---|------|------|
| 5.1 | **深度路由** | query 分析：宽泛 → 综述，具体 → 细节 |
| 5.2 | **跨域关联** | 搜索结果 ↔ wiki 段落 反向关联 |
| 5.3 | **知识边界感知** | 所有结果相似度 <0.3 → 诚实说没有 + 给外查建议 |

## 沈念 — 2 个 Phase

### Phase A ── 知识联动升级

> 依赖资料管理员 Phase 2（有 extracted_insight 和碰撞报告才能传）。

| # | 任务 | 改动 |
|---|------|------|
| A.1 | **接 librarian 处理结果** | `check_knowledge()` 不再读原始 tagged.json，改为读 librarian 产出的 `curated/insights/` 和碰撞报告 |
| A.2 | **输出变聪明** | 从 "今天有篇 #agent 文章要看看吗" → "今天 agent 记忆方向有个新方案 MemOS，和你之前关注的 LangMem 解决同一问题但省 35% token，要展开吗" |
| A.3 | **上下文携带** | 推送时附带你相关 wiki 的引用——"你之前写的记忆架构.md 里提到的三段式设计，这个新方案刚好补了第三段的实现" |

### Phase B ── 画像积累

> 不依赖资料管理员，沈念自身持续优化。

| # | 任务 | 改动 |
|---|------|------|
| B.1 | **about-me 质量提升** | 从对话中提取的不只是事实，还包括判断模式——"他选方案时优先考虑轻量而非企业级" |
| B.2 | **主动性增强** | 不只早晚和采集后触发，长对话后、搜索过但没结果时也能触发关怀 |
| B.3 | **输出风格适配** | 根据 about-me/偏好.md 调整回复的详细程度、技术深度、语气 |
