## 向量化机制

### 索引分层（替代单一 embeddings.json）

```
┌─────────────────────────────────────────────────────────┐
│                    Layer 1: 核心知识索引                   │
│                    core_index.json                       │
│                                                         │
│  内容: wiki + 综述 + 踩坑 + 关于我                         │
│  粒度: 按 ## 段落分块，每块独立向量                        │
│  文本: 段落全文 (不限 500 字)                              │
│  更新: wiki 内容变更时增量更新对应分块                      │
│  权重: 检索时优先 (你的知识体系，理应最容易被搜到)            │
│  预估: ~50 条 × ~15 段落 = ~750 条向量                    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    Layer 2: 近期资讯索引                    │
│                    recent_index.json                     │
│                                                         │
│  内容: 入库 < 7 天的每日AI精华 (筛选后)                     │
│  粒度: 每篇独立向量                                       │
│  文本: title + one_liner + extracted_insight (萃取要点)    │
│  更新: 每天增量追加新文章，移除到期归档的                     │
│  权重: 查询时加权 1.0 (常态), 时间越新权重越高               │
│  预估: ~50-80 条                                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    Layer 3: 归档索引 (可选)                │
│                    archive_index.json                    │
│                                                         │
│  内容: 7 天前归档的文章 (不参与默认检索)                     │
│  粒度: 每篇独立向量                                       │
│  更新: 归档时追加, 技术替代时标记为 stale                   │
│  权重: 仅 --include-archive 时检索                        │
│  预估: 随天数增长                                         │
└─────────────────────────────────────────────────────────┘
```

### 向量化频率

```
触发条件                          操作                          影响范围
──────────────────────────────────────────────────────────────────────
每日管线 (7:30)                  新文章精华编码 → recent_index       增量 8-12 条
每日管线 (7:30)                  到期文章从 recent_index 移除        删除 ~25 条
每日管线 (7:30)                  归档文章编码 → archive_index         增量 ~25 条

wiki 文件变更 (git hook/手动)    变更段落重新编码 → core_index        增量 1-N 段
策展生成新综述                    全文分块编码 → core_index            增量 ~15 段

技术替代标记                      旧条目在 archive_index 标 stale     元数据更新
成熟度迁移                        归档文章重新评估 → 可能召回          元数据 + 移动

模型升级 (手动触发)               全量重建三层索引                    全量
```

### 对比现在的向量化机制

| 维度 | 现在 | 升级后 |
|------|------|--------|
| 索引结构 | 单层 242 条 | 三层分层，核心/近期/归档 |
| 更新方式 | 每天全量重建 | 增量追加 + 到期移除 |
| wiki 粒度 | 整篇 500 字 | 按段落分块，全文编码 |
| 文章文本 | title + one_liner + summary | title + one_liner + **萃取要点** |
| 过期处理 | 无，永远留存 | 7 天从近期索引移除，降级到归档层 |
| 技术替代 | 无感知 | 归档条目标 stale，召回时降权 |
| 重建耗时 | ~3 秒 (全量 242 条) | <1 秒 (增量 8-12 条)，wiki 变更毫秒级 |

## 实现路线图

### Phase 1 ── 基础设施重构 (地基)

> 不做这一步，后面的入库关口和知识演化都挂在旧架构上

| # | 任务 | 改什么 | 关键逻辑 |
|---|------|--------|---------|
| 1.1 | **索引分层** | 新建 `index_manager.py` 替代 `index_builder.py` | 维护三个 JSON 文件：`core_index.json` (wiki/综述/踩坑)、`recent_index.json` (<7天精华)、`archive_index.json` (归档文章)。每个索引独立加载、独立检索 |
| 1.2 | **增量向量化** | 改 `index_builder.py` | 只编码当天新增 `.md`，追加到对应索引。旧条目用 id 匹配跳过。不再全量重建 |
| 1.3 | **段落级分块** | `index_manager.py` 的 wiki 编码器 | 非 article 笔记按 `## ` 分割为独立段落块，每块独立向量化。id 格式：`wiki::记忆架构::三段式记忆设计` |
| 1.4 | **归档机制** | 新建 `archiver.sh` | 每天检查 `raw/每日AI动态/`，>7天的 `.md` 移到 `archive/`，从 `recent_index` 移除，追加到 `archive_index` |
| 1.5 | **检索适配三层索引** | 改 `search_semantic.py` + `hybrid_merge.py` | Layer 1 (core) 权重 1.5 → Layer 2 (recent) 权重 1.0 → Layer 3 (archive) 默认不参与。语义分 × 层级权重 = 最终语义分 |

**改动文件：** `index_builder.py`(重构), `index_manager.py`(新), `archiver.sh`(新), `search_semantic.py`(改), `hybrid_merge.py`(改), `search_config.sh`(改配置)

### Phase 2 ── 入库关口 (每天 7:30 用的)

> 这是你每天接触的第一环——推送质量决定你对整个系统的信任

| # | 任务 | 改什么 | 关键逻辑 |
|---|------|--------|---------|
| 2.1 | **质量评分** | 改 `fetch_and_tag.sh` 的打标签 prompt | Claude 输出加 `quality: 1-5` 字段。标准：≤2 = 刷榜/重复/无关 → 不存。≥3 = 有价值 → 入库。5 = 突破性新范式 → 标记为重点追踪 |
| 2.2 | **去重合并** | 同上 prompt | 同一项目 24h 内多篇文章 → 合并为一条，one_liner 概括"发布了插件体系 (charts + sprites + core)" |
| 2.3 | **信号/噪声分类** | 同上 prompt | 加 `signal_type` 字段：`new_paradigm` / `incremental_update` / `rehashing_existing` / `benchmark_only`。只有前两类进入精华通道 |
| 2.4 | **萃取 extracted_insight** | 同上 prompt | 每条加 `extracted_insight` 字段 (50-80 字)：这条对你意味着什么。替代现在的 summary 作为向量化文本和推送内容 |
| 2.5 | **入库写入** | 改 `write_to_obsidian.sh` | 只写 quality ≥3 的文章到 `.md`文件。写入格式加 `extracted_insight`、`signal_type` 字段 |

**改动文件：** `fetch_and_tag.sh`(改 prompt), `write_to_obsidian.sh`(改), `获取信息/data/` 的 JSON schema(加字段)

### Phase 3 ── 知识演化引擎

> 知识库从"只增不减的仓库"变成"会自我更新的有机体"

| # | 任务 | 改什么 | 关键逻辑 |
|---|------|--------|---------|
| 3.1 | **追踪线程** | 新建 `thread_tracker.py` | 同一技术关键词连续 ≥3 天出现在新文章中 → 创建追踪线程。每天增量：当天文章命中线程关键词 → 追加到线程记录。线程管理存在 `threads.json` |
| 3.2 | **碰撞检测** | 新增 `collision_detector.sh`，在管线步骤 [4] 调用 | 每天精华入库后，Claude 对比新文章 vs 核心知识库 (wiki/综述) 的关键结论。输出三类标记：🔄冲突 (新证据可能推翻旧结论)、➕补充 (可以扩展某篇 wiki)、⚡替代 (新技术在同等场景下指标优于旧方案) |
| 3.3 | **成熟度自动迁移** | 新 `maturity_migrator.py` | 追踪线程中，按时间线升级：首次出现=🌱，连续7天有进展=🌿，连续30天稳定讨论=🌳。迁移时更新 `timeline.json` |
| 3.4 | **演化时间线** | 新建 `timeline.json` | 记录每个技术领域的演化轨迹：`"agent-memory": ["2025-03: LangMem (🌿)", "2025-09: MemOS (🌱)", "2026-05: MemOS 自进化 (🌿)"]` |
| 3.5 | **智能策展触发** | 改 `curator.sh` | 触发条件从固定 `≥15篇 + ≥14天` 改为：追踪线程活跃 (>3天连续) → 提前策展 / 成熟度迁移发生 → 触发 / 碰撞检测到替代 → 立刻标记旧条目标 stale |
| 3.6 | **wiki 更新提醒** | 碰撞检测的副产品 | 输出碰撞报告时，附带具体建议："建议更新 wiki/记忆架构.md 的第 3 段，新增 MemOS 的 hybrid-retrieval 方案" |

**改动文件：** `thread_tracker.py`(新), `collision_detector.sh`(新), `maturity_migrator.py`(新), `timeline.json`(新数据), `curator.sh`(改触发逻辑), `process_new.sh`(改调度)

### Phase 4 ── 主动输出

> 从"你让它查"到"它觉得你应该知道"

| # | 任务 | 改什么 | 关键逻辑 |
|---|------|--------|---------|
| 4.1 | **异常预警** | `process_new.sh` 加统计步骤 | 每日对比：相关文章数 vs 7日均值 (骤降 → 预警)、标签分布变化 (某个领域突然爆发或沉寂)、追踪线程有无停滞 |
| 4.2 | **盲区提醒** | 新增 `blindspot_check.sh` | 统计每个标签下你实际的查看/搜索次数。标签下有 >30 篇文章但你的交互次数为 0 → "这个领域你存了很多但从未关注" |

### Phase 5 ── 对话增强

> 你搜的时候，它不只是返回 8 条 Markdown

| # | 任务 | 改什么 | 关键逻辑 |
|---|------|--------|---------|
| 5.1 | **深度路由** | 改 `search_obsidian.sh`，检索前加一步判断 | query 分析：宽泛问题 ("agent 记忆怎么做") → 优先返回综述/全景图/路线图。具体问题 ("MemOS hybrid retrieval 实现") → 优先返回代码级/细节文章。由 Claude 在检索前判断 |

```
  └──→ Phase 2 (入库关口)
         └──→ Phase 3 (知识演化)
                └──→ Phase 4 (主动输出)
                       └──→ Phase 5 (对话增强)

Phase 1 是所有后续 Phase 的硬依赖 (索引不分层，后面都白做)
Phase 2 是每天管线的基础 (不过这一关，后面没数据)
Phase 3 是核心智能 (真正区别于"搜索引擎"的能力)
Phase 4+5 是用户体验层 (可并行开发)
```

## 关键决策点

| 环节 | 谁做决策 | 决策依据 |
|------|---------|---------|
| 去重合并 | Claude (规则明确) | 同一项目名 + 同日出现 |
| 信号/噪声 | Claude (需判断) | 新颖性、与你部署栈的相关性、是否刷榜 |
| 技术替代 | Claude (需对比) | 解决同一问题 + 有可比较的量化指标 |
| 成熟度迁移 | 规则 (可自动化) | 连续 N 天出现 + 文章数阈值 |
| 归档 | 规则 (可自动化) | 入库 > 7 天 + 非持续追踪中 |
| 盲区提醒 | 规则 (可统计) | 标签下文章数 vs 你的查看次数 |
| 深度路由 | Claude (需判断) | 查询词的具体程度 (怎么做 vs 实现细节) |
